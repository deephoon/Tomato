# Tomato 현재 수정사항 평가 보고서

작성일: 2026-06-25  
대상 저장소: `Tomato`  
평가 기준: 실제 배포 가능성, 기능 안정성, 인증/데이터 안전성, 사용자 혼동 가능성, 유지보수성, 검증 수준

---

## 1. 종합 판단

현재 Tomato는 **무료 정적 배포 기반의 MVP 서비스로는 배포 가능한 상태**까지 올라왔습니다. GitHub Pages 배포 파이프라인, Supabase 기반 인증/동기화, 주요 타이머 플로우, 작업 삭제/중복 방지, 아카이브 UI 정리까지 핵심 리스크가 여러 차례 줄었습니다.

다만 아직 "빈틈없는 서비스"라고 부르기에는 이릅니다. 가장 큰 남은 빈틈은 **브라우저 실제 사용 플로우 자동 검증 부족**, **Supabase 외부 설정 의존성**, **AI/백업/PWA 같은 기능 기대치와 실제 구현 사이의 차이**, **동기화 충돌 처리의 단순함**, **큰 `main.js`/CSS로 인한 유지보수 리스크**입니다.

현실적인 결론은 다음과 같습니다.

- **현재 상태**: 제한된 사용자에게 공개 테스트 가능
- **바로 공개해도 되는 범위**: 로그인, 오늘 작업 생성, 집중/휴식, 기록 확인, 기본 동기화
- **공개 전 반드시 더 봐야 할 범위**: 실제 브라우저 E2E, Supabase 인증 설정 재확인, 여러 기기/탭 동시 수정, 모바일/작은 화면, PiP 미지원 브라우저 안내
- **아직 광고하면 안 되는 표현**: "AI가 해준다", "완전 오프라인 지원", "모든 브라우저 위젯 지원", "데이터 백업 기능 제공"

## 3. 잘된 점

### 3.1 배포 경로가 실제로 작동한다

GitHub Pages + GitHub Actions 기반 배포가 이미 연결되어 있고, `main` 브랜치 push 후 `npm test` → `npm run build` → Pages 배포가 자동으로 실행됩니다. 이는 지금 단계에서 매우 중요한 진전입니다.

좋은 이유:

- 별도 서버 비용 없이 무료 배포 가능
- 배포 전에 테스트와 빌드가 자동으로 걸림
- 정적 SPA 구조와 잘 맞음
- Supabase를 백엔드로 분리했기 때문에 프론트 배포가 단순함

현재 배포 URL:

```text
https://deephoon.github.io/Tomato/
```

최근 성공한 배포:

```text
https://github.com/deephoon/Tomato/actions/runs/28084058662
```

### 3.2 이메일 인증 흐름을 더 안전하게 막았다

최근 수정에서 이메일 로그인/회원가입 흐름이 꽤 좋아졌습니다.

잘된 부분:

- 이메일 형식과 비밀번호 규칙을 Supabase 호출 전에 검증
- Supabase Confirm email 설정이 꺼져 있으면 앱이 인증 사용을 차단
- 회원가입 직후 세션이 생기더라도 즉시 로그아웃하여 "메일 확인 전 자동 로그인"을 막음
- 미확인 이메일로 로그인되면 즉시 로그아웃 처리
- 과거 로컬 fallback 인증 파일을 제거하여 "이메일처럼 보이지만 실제 소유 확인 없는 로그인" 문제를 줄임

이건 사용자 제보의 "이메일 로그인 뚫림" 위험에 직접 대응한 좋은 수정입니다.

남은 주의점:

- Supabase Dashboard의 Confirm email 설정은 앱 코드가 강제로 켤 수 없습니다.
- 앱은 잘못된 설정을 감지해 막을 수 있지만, 운영자가 Supabase 설정을 바꾸면 다시 점검해야 합니다.

### 3.3 작업 삭제/중복 문제의 원인을 제대로 줄였다

작업 영구 삭제 후 다시 들어오면 항목이 늘어나던 문제는 단순 UI 문제가 아니라 **로컬 ID와 Supabase UUID의 불일치**가 핵심이었습니다.

잘된 수정:

- 로컬 `t_...` 형식 ID를 Supabase UUID로 정규화
- 저장 시 현재 로컬 목록에 없는 원격 작업을 삭제
- 원격에 이미 생긴 중복 작업을 UI 반환 전에 signature 기반으로 dedupe
- 빈 작업 목록 저장 시 원격 작업 전체 삭제 케이스 테스트 추가

이 수정은 증상만 지운 것이 아니라 데이터 모델의 불일치를 맞춘 쪽이라 의미가 큽니다.

주의할 점:

- `saveTasks()`가 "현재 로컬 목록이 진실"이라는 방식으로 동작합니다.
- 여러 기기에서 동시에 수정하면 한쪽의 오래된 로컬 목록이 다른 기기의 작업을 지울 가능성이 있습니다.
- 공개 서비스 수준으로 가려면 `updated_at`, 버전, 충돌 해결 전략이 필요합니다.

### 3.4 사용자 혼동을 줄인 UI 정리가 좋다

최근 피드백 반영에서 다음 UI 변경은 방향이 좋습니다.

- 아카이브 하단 JSON 복구 파일 버튼 제거
- `[ AI 세분화 ]`를 `[ 작업 나누기 ]`로 변경
- `[ 영구 폐기 ]`를 `[ 삭제 ]`로 변경
- 일시정지 상태에서 오각형 회전/점멸을 멈춤

특히 JSON 복구 파일 버튼 제거는 중요합니다. 사용자가 내려받은 `.json` 파일을 "코드가 튀어나왔다"고 느끼면 서비스 신뢰도가 떨어집니다. 기능적으로는 백업일 수 있어도, 공개 UI에 노출하기에는 아직 설명 비용이 큽니다.

### 3.5 테스트 범위가 빠르게 좋아졌다

현재 테스트는 18개 파일, 90개 케이스입니다. 전부 E2E는 아니지만, 핵심 순수 로직과 저장소 어댑터에 테스트가 많이 붙었습니다.

현재 커버하는 주요 영역:

- 인증 입력 검증
- Supabase auth hardening
- 작업 ID 정규화
- 작업 저장/삭제 동기화
- 중복 작업 dedupe
- 타이머 시작/일시정지/재개/완료
- 세션 복원
- 아카이브 쿼리
- 통계 계산
- 작업 나누기
- 공개 UI 표면에 혼동 문구가 남지 않는지
- i18n 관련 회귀

이 방식은 지금 코드베이스에 맞는 좋은 전략입니다. DOM 전체를 억지로 테스트하기보다, 서비스/저장소 레이어를 먼저 잘게 검증하는 방향이 맞습니다.

### 3.6 성능 방향이 현실적이다

Three.js와 GSAP를 초기 번들에서 분리한 것은 맞는 방향입니다. 현재 빌드 기준으로 메인 JS gzip이 약 81 kB 수준이고, 무거운 3D 코드는 별도 청크로 분리되어 있습니다.

좋은 점:

- 초기 로딩 부담을 줄임
- 3D 실패가 앱 전체 실패로 번지지 않도록 지연 로드
- PiP 위젯은 사용자 제스처 요구 때문에 정적 import로 유지한 판단이 타당함

남은 점:

- `three-scene` 청크 자체는 큼
- CSS와 `main.js`가 커서 유지보수 리스크가 있음
- 실제 저사양 모바일에서 체감 성능은 아직 별도 검증이 필요함

### 3.7 제품 톤 정리가 좋아졌다

초기에는 기능명이 강한 세계관 표현에 치우쳐 실제 사용자에게 의미가 모호해질 수 있었습니다. 최근 변경으로 "작업 나누기", "삭제"처럼 사용자가 바로 이해할 수 있는 표현이 늘었습니다.

이 방향은 계속 유지하는 것이 좋습니다.

권장 원칙:

- 내부 세계관 표현은 분위기를 만드는 데만 사용
- 위험하거나 되돌리기 어려운 행동은 일반어 사용
- AI가 아닌 기능에는 AI라고 붙이지 않기
- 데이터 파일, 복구, 계정 같은 민감 영역은 설명 없이 노출하지 않기

## 4. 부족한 점과 리스크

### 4.1 실제 브라우저 E2E 테스트가 없다

가장 큰 빈틈입니다. 현재 테스트는 강하지만 대부분 단위/서비스/저장소 테스트입니다. 실제 사용자는 DOM, 라우팅, 폼 입력, Supabase 세션, 브라우저 API, 모바일 viewport를 함께 탑니다.

아직 자동 검증이 부족한 실제 플로우:

- 회원가입 → 이메일 확인 후 로그인
- 로그인 후 새로고침
- 오늘 작업 생성 → 수정 → 삭제
- 집중 시작 → 일시정지 → 재개 → 완료 → 휴식 → 대기
- 아카이브 기록 확인
- 다른 탭에서 상태 변경 후 반영
- 모바일 크기에서 버튼/텍스트 겹침 확인
- PiP 지원 브라우저와 미지원 브라우저 분기

권장:

- Playwright를 도입해 최소 happy path 3개부터 자동화
- Supabase 실계정 E2E는 staging 프로젝트 또는 테스트 계정으로 분리
- 스크린샷 기반 smoke test를 desktop/mobile 각각 추가

### 4.2 Supabase 외부 설정 의존성이 크다

앱 코드가 좋아져도 Supabase Dashboard 설정이 잘못되면 인증이 막히거나, 의도와 다르게 자동 확인될 수 있습니다.

특히 중요한 설정:

- Email provider enabled
- Confirm email enabled
- Site URL: `https://deephoon.github.io/Tomato/`
- Redirect URL: `https://deephoon.github.io/Tomato/**`
- RLS 정책 적용
- GitHub Secrets: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`

현재 앱은 `mailer_autoconfirm`이 켜져 있으면 차단하도록 되어 있습니다. 좋은 방어지만, 운영 관점에서는 배포 전 체크리스트가 필요합니다.

권장:

- `docs/deploy-checklist.md` 또는 README에 Supabase Dashboard 스크린샷/체크리스트 추가
- 배포 후 `/auth/v1/settings` 확인 절차 문서화
- 운영용 Supabase와 개발용 Supabase 분리

### 4.3 AI 기능 기대치가 실제 구현보다 커질 수 있다

현재 "작업 나누기"는 로컬 휴리스틱입니다. 이 자체는 MVP로 충분히 좋지만, 사용자에게 "AI가 해준다"로 보이면 기대치가 커집니다.

현재 상태:

- 작업 나누기: 로컬 결정론적 휴리스틱
- 아카이브 인사이트: Supabase Edge Function `generate-insight` 호출 시도 후 실패하면 로컬 fallback
- 실제 LLM 연동: 아직 운영 기능으로 보장되지 않음

최근 UI에서 "AI 세분화" 문구를 제거한 것은 잘한 결정입니다.

권장:

- 실제 LLM을 붙이기 전까지는 "자동 나누기", "추천", "작업 나누기" 정도로 유지
- Edge Function을 배포하지 않을 거면 아카이브 인사이트 UI도 과장하지 않기
- LLM 도입 시 비용, rate limit, 개인정보, prompt injection 방어를 별도 설계

### 4.4 동기화 충돌 전략이 아직 단순하다

현재는 단일 사용자, 단일 주 기기에서는 충분히 작동합니다. 그러나 여러 탭/기기에서 동시에 수정하면 충돌 가능성이 있습니다.

위험 예:

- A 기기에서 작업 1개 추가
- B 기기가 오래된 목록으로 `saveTasks()` 실행
- 서버에는 B의 목록 기준으로 A의 새 작업이 삭제될 수 있음

현재 `saveTasks()`는 "로컬 현재 목록에 없는 원격 row 삭제" 방식이라 삭제 버그를 고치는 데는 효과적이지만, 협업/다중기기 관점에서는 공격적입니다.

권장:

- 작업별 `updated_at` 비교
- tombstone 기반 삭제 기록
- `saveTasks()`를 전체 동기화가 아니라 create/update/delete 명령 단위로 전환
- 최소한 원격 fetch 기준 timestamp가 오래됐으면 전체 삭제를 막는 guard 추가

### 4.5 오프라인 큐가 사용자에게 보이지 않는다

오프라인 쓰기 큐가 있지만, 사용자는 현재 저장이 지연됐는지, 실패했는지, 재시도 중인지 알기 어렵습니다.

부족한 점:

- 오프라인 상태 배너 없음
- 동기화 실패 표시 없음
- 큐 처리 실패 시 사용자 피드백 부족
- 충돌 발생 시 해결 UI 없음

권장:

- 상단 meta bar에 `SYNCING`, `OFFLINE`, `SAVE FAILED` 상태 표시
- 실패한 명령 수 표시
- 재시도 버튼 또는 자동 재시도 안내

### 4.6 `main.js`와 `style.css`가 너무 크다

현재 구조는 빠르게 기능을 붙이기에는 좋았지만, 장기 유지보수에는 부담이 큽니다.

리스크:

- `main.js`가 렌더링, 이벤트 바인딩, 플로우 제어를 많이 갖고 있음
- 작은 UI 수정이 다른 화면에 영향을 줄 수 있음
- DOM 기반 함수가 많아 테스트하기 어려운 영역이 남음
- CSS가 커져 특정 클래스 변경의 영향 범위 파악이 어려움

권장:

- `main.js`에서 화면 단위 모듈 분리
- `focusView`, `plannerView`, `archiveView`, `authView` 식으로 책임 나누기
- DOM 테스트가 필요한 부분은 작은 view helper로 추출
- CSS도 화면별 파일 또는 명확한 섹션 책임 강화

### 4.7 README가 일부 최신 상태와 불일치한다

현재 README에는 아래 값이 오래된 상태입니다.

- 테스트: README에는 66 tests로 남아 있으나 현재는 90 tests
- i18n: README에는 207키로 남아 있으나 현재는 en/ko 각 195키

이건 기능 버그는 아니지만, 배포 준비 문서로는 신뢰도를 떨어뜨립니다.

권장:

- README의 수치형 내용은 자주 바뀌므로 "현재 테스트 수"를 직접 박기보다 검증 명령 중심으로 쓰기
- 정확한 수치를 쓸 경우 배포 전마다 업데이트

### 4.8 GitHub Actions Node 20 경고가 있다

최근 Actions 로그에서 Node.js 20 관련 deprecation annotation이 있었습니다. 현재 배포는 성공하지만, 장기적으로는 workflow의 Node 버전을 업데이트하는 편이 좋습니다.

권장:

- `.github/workflows/web-deploy.yml`의 `node-version: 20`을 Node 24로 올리는 것을 검토
- Vite/Vitest가 Node 24에서 문제 없는지 로컬/CI 확인

### 4.9 PWA/오프라인 앱으로 오해될 수 있다

현재 README에도 명시되어 있듯이 PWA는 정리되어 있고 일반 웹앱입니다. 하지만 "오프라인 큐"와 "위젯"이 있어 사용자는 설치형/오프라인 앱을 기대할 수 있습니다.

권장:

- 현재는 "웹앱"으로 명확히 안내
- PWA를 넣을 경우 manifest, icons, service worker, hashed asset precache까지 한 번에 처리
- 부분적인 sw만 추가하지 않기

### 4.10 관측성/운영 모니터링이 없다

현재는 로컬 콘솔 오류와 테스트 중심입니다. 실제 사용자에게 배포하면 다음을 알기 어렵습니다.

- 로그인 실패가 몇 건인지
- Supabase 저장 실패가 얼마나 나는지
- PiP 미지원 브라우저 비율이 어느 정도인지
- 특정 화면에서 JS 오류가 나는지
- 사용자가 작업 삭제/수정에서 막히는지

권장:

- 최소한 Sentry 같은 클라이언트 오류 수집 도입 검토
- 개인정보를 보내지 않는 선에서 주요 실패 이벤트만 기록
- Supabase 실패 시 사용자 메시지와 개발자 로그를 분리
